High-Level Design

Adaptive Inference
Router

A smart gateway that protects your GPU inference backend by classifying traffic into tiers and shedding excess load — before it causes damage.

What & Why

What is this?

A Python FastAPI gateway that sits in front of vLLM and makes smart routing decisions based on real-time backend health metrics.

Without it, every request hits vLLM regardless of load. Under pressure, everyone suffers equally — latency spikes, timeouts, crashes. With the router, we protect the system by shedding excess traffic before it causes damage.

💡

Analogy: Anthropic serves Free and Pro users on the same GPU fleet. Under heavy load, they throttle free-tier to protect Pro SLOs. Our router does the same — protect admitted requests by rejecting overflow.

Key insight: The router does NOT query metrics on every request. A background poller hits vLLM's /metrics every 3 seconds and caches the results in Redis. Each incoming request reads the cache — zero added latency on the hot path. If the poller dies, we fall back to GREEN (fail-open, not fail-closed).

Traffic Priority

Who gets served when things go RED?

When the router enters RED state, it doesn't reject everyone. Think of it like a queue: existing in-flight users are protected, new incoming requests are shed. The priority is always to finish serving what we already admitted before accepting more.

🎯

In Anthropic's case, it's paid vs free. In ours, it's existing vs new — requests already being processed get full resources, new requests get a fast 503 with a Retry-After header so they can come back shortly.

What the Router Watches

Cached Metrics from vLLM

The background poller grabs these from vLLM's /metrics endpoint and writes them into the Redis cache every 3 seconds.

num_requests_waiting
Queue Depth — how many requests are waiting to be processed. This is the primary signal. A growing queue means vLLM is falling behind.
gpu_cache_usage_perc
KV Cache Fill % — how full the GPU's key-value cache is. When this hits ~90%+, latency spikes hard because vLLM can't allocate memory for new sequences.
time_per_output_token
Inter-token Latency — time between each generated token. This is what the user "feels" as typing speed. Higher = sluggish responses.
Decision Logic

Three Tiers of Traffic

🟢 GREEN

Healthy
  • Queue depth < 5
  • KV cache < 70%
  • Forward to vLLM
  • max_tokens: 512
  • Normal full service

🟡 YELLOW

Degraded
  • Queue depth 5–9
  • OR KV cache ≥ 70%
  • Forward to vLLM
  • max_tokens: 128
  • Shorter answers, still served

🔴 RED

Load Shedding
  • Queue depth ≥ 10
  • OR KV cache ≥ 90%
  • OR token time > 100ms
  • Return 503 instantly
  • Retry-After: 5s
❓

Why three tiers, not two? Two tiers means serve or reject — that's harsh. Three tiers gives us graceful degradation. YELLOW says "I'll still help you, just shorter." This is what Anthropic does: free users get shorter/slower responses before getting cut off entirely. It's a better user experience.

SLO Engineering

SLO Targets

503s from load shedding are intentional. They protect the SLO. The real question is: of requests we chose to serve, did we serve them well?

SLO Target
P95 latency (admitted requests) < 15 seconds
Error rate (non-503 errors) < 1%
Intentional shed rate < 20% of total traffic
Autoscaling

SLO-Driven Autoscaling

A second GPU worker node sits pre-joined in the cluster but tainted (NoSchedule) — no pods run on it. When the router detects sustained SLO breaches, a control loop removes the taint, a new vLLM pod schedules, and traffic rebalances.
